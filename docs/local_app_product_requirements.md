# ETF Quant 本地 App 产品需求

## 背景

使用者每天早上打开本地 app，核心问题不是“策略参数是什么”，而是：

- 今天要不要调仓。
- 如果要调仓，买什么、卖什么、各买卖多少。
- 调仓后仓位应该长什么样。
- 使用这套策略之后，实盘组合每天收益率如何，和回测表现是否持续一致。

当前 app 已经具备策略选择、持仓录入、行情刷新、调仓计划生成、回测曲线展示等基础能力，但仍偏工程工具。用户需要在多个区块里自行拼接结论，且缺少实盘收益跟踪账本。

## 当前功能不全

### 用户视角

- 首页没有明确的“今日结论”。用户生成计划后仍要自己判断交易动作是否重要、是否可以保持不动。
- 调仓动作只列订单，没有同步展示调仓后的目标仓位和现金变化，执行前信心不足。
- 当前持仓会被生成计划后的预测持仓覆盖，但缺少清晰提示，用户容易误以为已真实成交。
- 回测曲线只能说明策略历史表现，不能回答“我实际跟着做之后每天收益如何”。
- 缺少最近一次生成计划的跟踪记录、组合净值、日收益率、累计收益率等实盘状态。
- 对数据日期、行情刷新状态、策略输出目录等信息已有展示，但没有转化成风险提示，例如数据是否滞后。

### 产品经理视角

- app 的核心使用场景应是“早晨决策台”，不是策略浏览器。
- 页面优先级需要从“配置和图表”调整为“今日操作、仓位、收益跟踪”。
- 需要形成闭环：录入真实持仓 -> 刷新行情 -> 生成计划 -> 执行交易 -> 保存成交后持仓 -> 次日查看收益。
- 需要有轻量级本地账本，记录每次生成计划时的策略、日期、组合市值、现金、日收益、累计收益。
- 收益跟踪应基于用户实际录入持仓和最新价格，不替代券商对账，但要能快速判断策略跟随效果。
- 需要把“预测持仓”和“真实持仓”区分清楚，避免误导用户。

## 目标

1. 早上打开 app，用户能在 30 秒内知道是否需要调仓。
2. 生成计划后，用户能看到清晰的买卖清单、目标仓位和预计现金。
3. app 自动记录当日组合市值和收益，形成可视化跟踪。
4. 保持本地优先，不引入账号登录、券商 API 或外部服务依赖。

## 本期需求范围

### 今日行动

- 展示计划日期、行情日期、刷新方式和交易笔数。
- 根据交易笔数给出一句明确结论：
  - 无交易：保持当前持仓。
  - 有交易：按金额排序处理买卖动作。
- 订单表保留估算价格、交易份额、估算金额、目标权重。

### 目标仓位

- 展示所有目标权重大于 0 的资产。
- 展示当前权重、目标权重、预计调仓后权重、预计份额。
- 展示预计现金，提醒这是基于估算价格和整手约束的结果。

### 实盘收益跟踪

- 每次生成计划时，按当前录入持仓、现金和最新收盘价计算组合市值。
- 对同一策略和同一交易日保留最新一条记录，避免重复生成造成多条脏数据。
- 计算：
  - 当日组合市值。
  - 上次记录以来日收益率。
  - 首次记录以来累计收益率。
  - 最近记录日期和记录数量。
- 页面展示最近跟踪曲线和最近几条收益记录。

### 数据与状态

- 本地跟踪文件默认保存到 `data/live/performance_log.csv`。
- 当前持仓继续保存到 `data/live/positions.csv`。
- 当前策略、整手数量、最小交易额、是否刷新行情继续保存到 `data/live/app_state.json`。

## 暂不做

- 不直接连接券商账户。
- 不自动确认真实成交。
- 不处理申购赎回、分红、转托管等复杂资金流水。
- 不引入多账户、多币种汇率换算。
- 不替代正式的净值归因系统。

## 验收标准

- 启动 app 后可正常读取策略、持仓、回测和收益跟踪。
- 生成计划后，页面同时更新今日行动、目标仓位、订单和收益跟踪。
- 重复生成同一策略同一日期计划时，收益日志只保留一条该日期记录。
- 测试覆盖本地收益日志写入、日收益率计算、bootstrap 返回跟踪数据。
